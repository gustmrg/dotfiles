# DevLog

Use the `devlog` CLI to persist development work notes and generate summaries. DevLog stores data under `~/.devlog/`; the CLI is the source of truth for durable entries and summaries.

Default to running DevLog CLI commands. Do not create, edit, or delete DevLog data files manually unless the user explicitly asks for manual file work. This includes files under `~/.devlog/entries/`, `~/.devlog/summaries/`, `~/.devlog/templates/`, and `~/.devlog/config.json`.

The agent controls the wording of entries. Persisted summaries are generated by the CLI from those entries, so entry quality is the main place to apply templates and style guidance.

Write DevLog entry descriptions in Brazilian Portuguese (`pt-BR`) by default, unless the user explicitly asks for another language. Keep project names, file paths, commands, package names, tags, and technical identifiers in their original form.

## Stable CLI Surface

Use only commands that are implemented in the current DevLog CLI unless the user explicitly asks to experiment with planned features.

```bash
devlog init
devlog add "<description>" -p <project> -t <tag1,tag2> --date <YYYY-MM-DD>
devlog list --date <YYYY-MM-DD>
devlog summary create --date <YYYY-MM-DD>
devlog summary show --date <YYYY-MM-DD>
devlog summary list --week
devlog summary list --month
devlog summary list --from <YYYY-MM-DD> --to <YYYY-MM-DD>
```

Do not use `devlog summary create --style ...` unless also using `--ai`; the CLI rejects `--style` without `--ai`. Treat template-based writing as entry guidance unless the installed CLI confirms support for styled summaries.

## Entry Workflow

When the user asks to add or insert an entry:

1. Infer the entry from the user's text and the current session context. Use concrete work that actually happened: files edited, commands run, bugs diagnosed, tests verified, blockers found, or docs written.
2. Use today unless the user gives a date. Dates must be `YYYY-MM-DD`.
3. Infer `project` from the user, current repository, or current working directory. If this is ambiguous and the project matters, ask one short clarification; otherwise let DevLog use its configured default.
4. Infer tags from meaningful work areas such as `frontend`, `backend`, `tests`, `docs`, `cli`, `config`, `bugfix`, `refactor`, or the stack/component involved. Omit tags if they would be noisy guesses.
5. Write the entry using the requested template. Default to `concise`. Use Brazilian Portuguese unless the user asks otherwise.
6. Run `devlog add` with the final description. Quote descriptions and shell arguments safely.
7. Report the entry text and command outcome. Do not claim an entry was saved if the command failed.

## Summary Workflow

When the user asks to create a summary:

1. Determine whether they want a persisted DevLog summary, a chat summary, or both.
2. If the summary should reflect the current session and no relevant entries exist yet, first create one or more entries from the session context, then run `devlog summary create`.
3. Use `devlog summary create --date <YYYY-MM-DD>` for persisted summaries. The current non-AI CLI creates a concise grouped Markdown summary from entries.
4. Use `devlog summary show --date <YYYY-MM-DD>` after creating a summary when the user wants to see it.
5. Do not manually create or edit summary files. If the user wants a different persisted summary format, explain that DevLog CLI support is needed unless they explicitly request manual file editing.

## Entry Templates

### Concise Entry

Use for quick daily logging and default agent-written notes.

```text
<Past-tense action verb> <specific work item> <important verification/status if needed>
```

Rules:

- Write one short sentence or sentence fragment.
- Aim for 8-18 words.
- Start with a concrete past-tense verb when natural.
- Include tests or verification only when they actually happened.
- Avoid background narrative and vague phrases like "worked on code".

Examples:

```text
Adicionou instrucoes do skill DevLog para entradas de sessao e resumos
```

```text
Diagnosticou suporte a estilos de resumo e documentou o comportamento estavel da CLI
```

### Detailed Entry

Use when the user wants more context, or the work has important scope, rationale, or verification state.

```text
<Past-tense action verb> <specific work>, including <key detail>, to <reason/outcome>; <verification or unresolved state>
```

Rules:

- Write one complete sentence.
- Aim for 20-40 words.
- Preserve purpose, key implementation detail, and verification state when known.
- Use multiple entries for distinct workstreams instead of one overloaded sentence.
- Do not invent tests, impact, or blockers.

Example:

```text
Criou um skill DevLog para agentes, incluindo fluxos seguros para entradas e resumos via CLI, para persistir o contexto da sessao de forma consistente; avaliacoes formais nao foram executadas.
```

## How Entry Style Affects Summaries

Because CLI summaries are built from entries, choose the entry template based on the summary the user will want later:

- Use `concise` entries when the user wants quick daily summaries with compact bullets.
- Use `detailed` entries when the user wants summaries that preserve purpose, implementation context, verification, or follow-ups.
- Split distinct workstreams into separate entries so CLI summaries can group and present them cleanly.
- Include project and tags whenever they are known, because they make later filtering and summary grouping more useful.

## Quality Bar

- Prefer durable outcomes over implementation minutiae.
- Preserve uncertainty. Use phrases like `investigated`, `prepared`, or `not yet verified` when work is incomplete.
- Do not add secrets, tokens, private customer data, or personal information to DevLog entries.
- Do not run destructive commands. DevLog logging should only initialize config, add entries, list entries, create summaries, or show summaries.
- Do not write directly to DevLog files by default. Use the CLI for all normal entry and summary operations.
- If `devlog` is not installed, say so and provide the exact command that would have been run after installation.
